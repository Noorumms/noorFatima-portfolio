## The Problem

Most beginner Android projects are single-screen demos with all logic crammed
into one Activity. I wanted to build apps that mirrored how production Android
apps are actually structured — with a clear separation between UI, state, and
data — while shipping something genuinely usable.

## What I Built

Two Android applications built with Jetpack Compose: an Affirmations app and
a Quotes app. Both use Firebase Authentication for user accounts and Firestore
for storing and syncing user data. Search is implemented as real-time
filtering backed by `StateFlow`, and both apps follow MVVM — Composables
observe state from ViewModels, which handle business logic and talk to
Firebase, keeping UI code free of data-fetching logic.

## Tech Stack

- **Kotlin** — the standard language for modern Android development
- **Jetpack Compose** — declarative UI that maps cleanly onto a
  state-driven (MVVM) architecture
- **Firebase Auth** — handles user sign-up/sign-in without building custom
  auth infrastructure
- **Firebase Firestore** — real-time NoSQL storage for user-specific data
- **MVVM** — keeps Composables dumb (rendering state) and ViewModels
  responsible for logic and data access

## Key Challenges

- **Real-time search without jank.** Filtering a list on every keystroke
  naively can cause UI stutter. Using `StateFlow` with a debounced search
  query meant the UI only recomposed when the filtered list actually changed.
- **State hoisting.** Getting Compose's unidirectional data flow right meant
  being disciplined about where state lived — ViewModels held the source of
  truth, and Composables only received state and event callbacks, never
  mutated state directly.
- **Firestore sync vs. local state.** Keeping the UI responsive while waiting
  on Firestore reads/writes required exposing loading and error states
  through the ViewModel, so the UI could show appropriate feedback instead of
  freezing.

## Results

- Two functioning apps with working authentication, persistent user data, and
  real-time search
- A consistent MVVM structure across both apps, making the codebase easy to
  navigate and extend
- Clean separation between UI (Composables), state (ViewModels/StateFlow),
  and data (Firebase) — each layer testable in isolation

## Lessons Learned

- MVVM with Compose isn't just a pattern to memorize — it directly solves
  real problems like recomposition performance and testability.
- Firebase removes a lot of backend work, but designing the Firestore data
  model still requires the same thinking as designing a SQL schema.
- Building two apps with the same architecture reinforced which parts of the
  pattern were genuinely reusable versus app-specific.
